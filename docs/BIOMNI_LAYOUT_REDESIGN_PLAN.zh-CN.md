# BioClaw 本地网页改版方案

## 目标

参考 `Biomni.png` 的工作台式布局，将当前 BioClaw 本地网页从“并排的聊天面板 + trace 面板”重构为“**项目工作台**”体验：左侧管理任务与文件，中间聚焦主要内容，右侧查看进度与结果，同时保留 BioClaw 现有的线程、聊天、上传、trace 等核心能力。`Workspace tree` 不再作为主界面常驻区域。

本方案先聚焦 **页面布局与信息架构**，不要求先改后端协议。目标是在尽量复用现有 API 和前端逻辑的前提下，让页面观感和交互方式尽可能接近 Biomni。

---

## 0. 根据评审修正后的实施原则

结合最新评审意见，本方案做以下明确修正：

- **四栏完整布局只在更宽屏幕启用**。完整工作台断点上调到 `1400px+`。`1100px~1399px` 之间不强行同时展示四栏，而是让右侧 inspector 退回切换式视图。
- **第一版不做 artifact-first 自动推断**。只提供手动切换，避免视图频繁跳转。
- **第一版不把 thread 真的改造成“有状态任务对象”**。界面上只做任务卡风格重绘，数据底层仍沿用现有 thread 结构。
- **PDF 预览采用浏览器原生能力**。优先用 `iframe` 加载 PDF/图片，不引入 PDF.js，也不要求后端新增预览接口。
- **Workspace tree 从主界面移除**。右侧只保留任务进度与结果文件列表；必要时未来再以折叠调试面板形式回归。
- **实际改动入口以当前代码为准**。当前前端确实由 `src/channels/local-web/assets/index.html`、`style.css`、`app.js` 提供，不按不存在的 `html-template.ts` 假设执行。

本次先做一个“稳的第一版”：先把主次关系、栏位结构、结果访问路径改对，再考虑更激进的 artifact-first 自动化。

## 1. 现状与问题

当前 BioClaw 页面由三块并排区域拼接而成：

- 左：`trace` 面板，包含追踪列表和 workspace tree
- 中：线程列表
- 右：聊天区域

从 `Bioclaw.png` 可以看到当前方案存在几个问题：

### 1.1 视觉层级太碎

- 顶部 header、左 trace、线程列表、聊天区都像独立小应用拼起来的
- 分隔线、卡片、按钮、pill 很多，但缺少一个真正的“主舞台”
- 页面第一眼看不到当前任务的主内容到底在哪里

### 1.2 主次关系反了

- trace、threads、chat 三块几乎同权
- 但对于 SEC 报告、图表分析、文件查看这类任务，用户真正关心的是“当前成果物”
- 当前页面没有一个明确的 artifact / report stage

### 1.3 顶部栏承担了过多临时信息

- Live、Connected、Settings 都堆在 header
- header 更像控制条，不像项目工作台的“项目上下文栏”
- 它没有帮助用户理解自己在做哪个任务、看哪个结果

### 1.4 trace 面板占位过重

- trace 作为调试 / 观测信息，本应是辅助信息
- 当前它和聊天区平级，占了一个完整大栏
- 对普通分析用户来说，这让页面显得技术感强但效率不高

### 1.5 聊天区缺少“工作区语义”

- 当前聊天区还是典型的 chat feed
- 而 Biomni 的中心区域是 report/document-first
- 对 SEC、报告、图片、PDF、结果文件类任务，BioClaw 也应该更偏向“工作台”而不是纯聊天窗口

## 2. Biomni 可借鉴的核心特征

从 `Biomni.png` 看，值得借鉴的不是表面颜色，而是它的 **信息架构**：

### 2.1 明确的三层工作台结构

- 最左是一条窄导航栏，只承担全局切换
- 第二列是项目/任务列
- 中间是唯一主内容区
- 右侧是状态与结果侧栏

### 2.2 中间区域绝对主导

- 中央文档区是整页视觉焦点
- 用户会自然理解：这里是当前任务真正的输出
- 左右两侧都在“服务”中间内容，而不是和中间抢注意力

### 2.3 状态信息靠右，不抢主内容

- 任务步骤、待办、结果文件放在右侧
- 属于辅助决策信息
- 能看，但不盖住主报告

### 2.4 左侧不是“聊天列表”，而是“项目上下文”

- 项目、任务、文件都与当前工作流有关
- 左侧给人的是 workspace / project 的感觉，而不是消息软件的联系人列表

### 2.5 整体更像生产工具，而不是 demo 界面

- 少量更坚决的边界线
- 更稳定的网格分区
- 更少漂浮感和胶囊感
- 更接近“桌面工作台”

## 3. 设计目标

BioClaw 改版后的页面，不追求 1:1 临摹 Biomni，而是要达到下面这几个结果：

### 3.1 目标一：把“当前任务成果”放到页面中心

中心区域应该优先展示：

- 当前对话中最新的重要消息
- 报告类任务的 PDF / 文档 / markdown 内容
- 图片、表格、文件结果

### 3.2 目标二：把 trace 从“主面板”降级成“右侧辅助面板”

trace 很重要，但不应该继续占据整块左半屏。

### 3.3 目标三：把 thread list 变成 project/task rail

线程依然保留，但视觉语义从“聊天列表”提升为“项目内任务列表”。

### 3.4 目标四：把 upload / workspace 变成工作台原生能力

不要把上传和下载只做成消息里的按钮，而是让用户在布局上自然理解：

- 文件属于当前任务
- 结果文件属于右侧 results 区
- 上传入口属于项目工作流的一部分

### 3.5 目标五：保留 BioClaw 的现有能力，不要求重写后端

优先复用现有接口：

- `/api/messages`
- `/api/threads`
- `/api/upload`
- `/api/trace/list`
- `/api/trace/stream`

这次先做前端结构升级，不引入大规模后端协议改造。

## 4. 推荐的目标布局

### 4.1 桌面版总体结构

建议改成 **四段式工作台**：

```text
┌──────┬──────────────────┬──────────────────────────────────────┬──────────────────────┐
│导航栏│项目/任务侧栏      │主内容区                              │状态/结果侧栏          │
│ 72px │ 280~320px        │ 自适应主舞台                         │ 320~360px            │
└──────┴──────────────────┴──────────────────────────────────────┴──────────────────────┘
```

注意：

- `1400px+`：完整四栏
- `1100px~1399px`：隐藏最左窄导航栏，右侧 inspector 不常驻
- `<1100px`：回退到更简单的切换式布局

对应关系如下：

| 目标区域 | 作用 | 当前能力映射 |
|---|---|---|
| 全局导航栏 | 全局入口与品牌 | Logo、项目、资源、帮助、设置 |
| 项目/任务侧栏 | 当前项目、线程列表、上传入口、近期文件 | 现有 thread rail + upload/workspace 摘要 |
| 主内容区 | 当前报告/聊天/文档的主视图 | 现有 chat main，升级为 artifact-first |
| 状态/结果侧栏 | trace 步骤、输出文件 | 现有 trace + 消息中的文件结果，重组为右侧 inspector |

### 4.2 导航栏

最左窄栏固定宽度 72px 左右，职责要非常明确：

- 顶部：BioClaw 标识
- 中部：少量一级导航 icon
- 底部：帮助 / 账户 / 设置

建议一级导航只保留 3 到 4 个：

- `项目`
- `资源`
- `追踪`
- `设置/帮助`

这条栏不展示业务细节，只负责“切换上下文类型”。

### 4.3 项目/任务侧栏

第二列承担 Biomni 左侧主栏的角色。建议拆成三段：

#### 顶部：项目头

- 标题：当前项目
- 一个 project selector 或当前任务名
- 新建任务按钮
- 搜索按钮

#### 中部：任务列表

把当前 `thread-list` 重新定义成任务列表：

- 每个 thread 以“任务卡片”形式显示
- 主标题更大
- 次信息显示更新时间
- 当前激活项用淡色高亮，而不是单纯边框
- rename / archive 做成右上角轻图标按钮

#### 底部：上传与最近文件

模仿 Biomni 左栏底部“云盘”区域：

- 固定一个 `上传` 动作
- 展示 1 到 5 个最近上传文件
- 点击文件时将该文件设为主舞台内容或右栏当前焦点

这样用户会更自然地理解“线程 + 文件 + 当前分析工作”属于同一项目空间。

### 4.4 主内容区

这是改版最关键的部分。主内容区必须从“聊天流”升级成“工作主舞台”。

建议拆成三层：

#### 顶部：上下文标题栏

显示当前任务标题，例如：

- `Protein SEC Analysis Report`
- 右侧可放 `分享`、`布局`、`切换视图`

这层不是全局 header，而是主内容自身的 title bar。

#### 中部：内容舞台

主内容舞台建议支持两种视图模式：

1. `Artifact-first`
   适合 PDF、图表、长报告、markdown 结果
2. `Conversation-first`
   适合普通聊天或尚未产出成果物的任务

第一版策略：

- 默认进入 `Conversation-first`
- 用户点击“结果”视图或点击文件项后，进入 `Artifact-first`
- 不做自动推断

##### Artifact-first 模式

中心优先展示：

- 当前最新的 PDF / 文档 / 结果摘要
- 图片网格
- 表格与长文 markdown

聊天消息不再一条条铺满整个区域，而是：

- 仅显示关键消息
- 或折叠成“过程记录”小节

##### Conversation-first 模式

保留当前消息流，但视觉上要更像文稿区：

- 减少 bubble 感
- bot 输出更接近 document block
- 文件卡片统一成结果块样式

#### 底部：固定悬浮输入区

参考 Biomni，把 composer 变成主内容区底部固定 dock：

- 输入框更宽、更扁
- 底部工具条内嵌上传、@、附加操作
- 发送按钮更像单个主动作，而不是一个大绿色 CTA
- plan mode / special toggles 放在右下次级控制区

### 4.5 右侧状态/结果侧栏

右栏建议分成上下两段：

#### 上段：任务进度 / trace inspector

把当前 trace 内容提炼成“任务步骤”视图，而不是完整事件流直接铺开。

推荐层级：

- `待办` / `进行中` / `已完成`
- 每个 step 为可折叠卡片
- 默认只展示摘要
- 展开后再看详细 trace 事件

这会更接近 Biomni 的 checklist 风格。

#### 下段：结果文件列表

将当前对话中出现过的结果文件整理成一个结果面板：

- 标题：`结果`
- 搜索框：过滤文件
- 文件列表按时间或类型排序
- 每项支持：
  - 打开
  - 下载
  - 设为主舞台
  - 定位到工作区

相比现在单纯显示 `Workspace tree`，这会更接近用户实际需要的“结果访问区”。

## 5. 视觉语言建议

Biomni 更像严肃的工作台，不像聊天 app。BioClaw 改版建议采用下面的视觉方向。

### 5.1 配色

建议从当前强烈的深色/高对比胶囊风，转向更克制的浅色工作台风：

- 主背景：微暖白或冷白，不要纯白
- 分隔线：细、清晰、低饱和
- 主强调色：保留 BioClaw 绿色，但减少大面积使用
- 状态色：仅用于连接状态、当前任务、高亮 step

### 5.2 圆角与阴影

建议减少现在的“卡片到处都有圆角”的感觉：

- 大面板：中小圆角
- 列与列之间更依赖边界线，不依赖阴影
- 阴影尽量轻

### 5.3 字体与排版

Biomni 的中心文档区更像阅读器。BioClaw 可以做双字体策略：

- UI：现代无衬线
- 文档/报告内容区：可考虑带一点书面感的 serif 或更稳定的阅读字体

但如果暂时不引入新字体，至少要做到：

- 标题更大、更稳
- 正文行高更舒展
- 表格更像文稿而不是聊天 markdown

### 5.4 控件风格

建议整体从“pill + bubble + shadow”转向：

- 更平的按钮
- 更清楚的边框
- 更少的漂浮元素
- 更稳定的栏位关系

## 6. BioClaw 到 Biomni 的功能映射

下面给出最重要的一一映射关系。

| 当前 BioClaw | 改版后位置 | Biomni 对应语义 |
|---|---|---|
| `thread-rail` | 左侧项目/任务栏中部 | Tasks |
| `newThreadBtn` | 左侧项目栏头部 | New task |
| `messages` | 中央主舞台 | Report / conversation |
| `composer-card` | 中央底部 dock | Ask / input bar |
| `timeline` | 右栏上半区 | Checklist / trace |

| `connPill` / `traceConnPill` | 缩到顶栏或状态角标 | Connection indicator |
| `settingsDrawer` | 保留为抽屉 | Settings |

## 7. 代码层面的实施方案

本次建议优先在这三个文件上动手：

- [src/channels/local-web/assets/index.html](../src/channels/local-web/assets/index.html)
- [src/channels/local-web/assets/style.css](../src/channels/local-web/assets/style.css)
- [src/channels/local-web/assets/app.js](../src/channels/local-web/assets/app.js)

### 7.1 `index.html` 结构改造

当前结构核心问题是：

- `panelTrace` 和 `panelChat` 还在做“双主面板”
- `thread-rail` 只是聊天列表，不是项目栏

建议新结构改为：

```text
workspace-shell
├── app-rail
├── project-sidebar
│   ├── project-header
│   ├── task-list
│   └── file-tray
├── workspace-main
│   ├── workspace-topbar
│   ├── workspace-stage
│   └── composer-dock
└── inspector-sidebar
    ├── inspector-progress
    └── inspector-results
```

保留现有 DOM id 的思路：

- 能复用现有 JS 查询的 id 就尽量复用
- 先重排 DOM，再逐步替换类名和语义
- 避免一次把所有逻辑事件绑定都打碎

### 7.2 `style.css` 需要重写的重点

建议不是在当前样式上补丁式叠加，而是针对 shell 级结构做一次明确重构。

重点包括：

- 新的四栏 grid
- 导航窄栏样式
- 项目栏卡片样式
- 中央主舞台排版
- 右侧 inspector 分区
- 新的底部 composer dock
- artifact-first 与 conversation-first 两套 stage 样式

尤其要处理下面几个现有问题：

- 顶部 header 占位过重
- trace 与 chat 的同级关系
- message bubble 太像聊天 app
- 文件卡片太像消息内附件，不像 workspace result

### 7.3 `app.js` 交互逻辑调整

JS 不必大改数据来源，但需要新增几个 UI 状态：

- `layoutMode = artifact | conversation`
- `selectedThread`
- `selectedArtifact`
- `selectedResultFile`
- `inspectorSectionState`

还需要新增或调整这些行为：

- 点击结果文件时，把它送入中央 stage
- 线程列表不再只是 render button，而是 render task card
- trace 视图支持摘要/展开两级展示

第一版不做：

- 自动 artifact 推断
- thread 状态机
- 专门的 PDF 渲染库接入

## 8. 关键交互方案

### 8.1 用户打开一个 SEC 任务

页面进入该任务后：

- 左栏高亮当前任务
- 中央显示报告主内容
- 右栏顶部显示当前生成步骤
- 右栏底部显示所有结果文件

### 8.2 用户点击某个 PNG/PDF 结果

- 中央 stage 切换为该文件预览
- 右栏结果列表高亮该文件
- 左栏任务不变

第一版预览策略：

- PDF：浏览器原生 `iframe`
- 图片：原生 `<img>`
- 其他文件：显示文件信息 + 打开/下载按钮

### 8.3 用户回看聊天过程

在主内容区上方或侧边提供一个 `Conversation` / `Report` 切换：

- `Report`：中心显示成果物
- `Conversation`：中心显示消息流

这样既保留 BioClaw 的聊天本质，又能靠近 Biomni 的 artifact-first 工作流。

### 8.4 用户查看 trace

默认只看步骤摘要：

- Extract / Analyze / Generate figures / Write report

展开后再看：

- 具体 run
- tool 调用
- error
- raw payload

这样右侧能保持 Biomni 式整洁感，而不是直接暴露太多底层调试信息。

## 9. 响应式策略

Biomni 风格适合桌面，但 BioClaw 仍要兼顾小屏。

### 9.1 大桌面（1400px+）

- 四栏完整展示
- 中央主舞台最大

### 9.2 中等桌面 / 平板（1100px~1399px）

- 隐藏最左窄导航栏
- 右栏 inspector 改成切换式或抽屉式
- 保留左任务栏 + 中央主舞台

### 9.3 手机

不建议强行维持 Biomni 四栏结构。手机上应改成：

- 顶部切换：任务 / 内容 / 结果
- 底部固定输入栏
- inspector 与 project rail 走 slide-over 模式

换句话说：

- 桌面学 Biomni
- 移动端保持 BioClaw 自己更合理的轻量布局

## 10. 分阶段实施建议

建议分四期做，而不是一次全改。

### Phase 1：工作台骨架

目标：

- 引入左窄导航栏
- 把 trace 主栏降级到右侧 inspector
- 把 thread rail 重命名并重设计为任务栏
- 让中央区域成为视觉主栏
- 移除主界面的 `Workspace tree`

产出：

- 页面第一次具备 Biomni 式骨架
- 主舞台最小宽度在完整四栏下不低于 `500px`
- trace 不再和聊天区同权

### Phase 2：主内容区 artifact-first

目标：

- 主舞台支持报告/图片/文件预览
- 文件点击后能在中间打开
- 聊天与成果物分离
- 只提供手动 `对话 / 结果` 切换

产出：

- 真正有“Biomni 感”的中心舞台
- PDF 文件能在主舞台内打开而不是新标签页
- 有明确的 artifact / conversation 切换按钮

### Phase 3：右栏结果与步骤整合

目标：

- trace 做摘要化
- 结果文件列表按时间排序
- 右栏更像结果 inspector，而不是调试输出容器

产出：

- 右栏从工程调试视图变成用户友好工作栏
- trace 默认折叠为摘要，展开后才看 raw 事件

### Phase 4：视觉打磨

目标：

- 调整留白、字号、边框、hover、滚动条、按钮
- 统一图标体系
- 提升 PDF / markdown / file 卡片的视觉一致性
- 压缩无用英文说明，默认中文优先

产出：

- 接近 Biomni 的整体完成度
- bot 输出视觉上更像文档块，而不是气泡聊天

---

## 11. 建议的最小可行版本

如果想先快速验证方向，建议第一版只做以下几件事：

1. 新增左侧窄导航栏
2. 把线程列表改成 Biomni 式任务栏
3. 把 trace 挪到右侧，并移除 `Workspace tree`
4. 中央聊天区去掉强烈 bubble 感，改成文档主舞台风格
5. composer 改成底部固定 dock

只做这五步，页面气质就会从“聊天工具”明显转向“分析工作台”。

---

## 12. 风险与注意事项

### 12.1 不要只改颜色，不改结构

如果只是把现在页面刷成 Biomni 的浅色配色，但仍保留：

- 左 trace
- 中间 thread
- 右 chat

那最后只会得到一个“浅色版旧布局”，不会有真正的 Biomni 感。

### 12.2 不要把 trace 完全隐藏

BioClaw 的一个独特优势就是可观测性。正确做法不是删除 trace，而是：

- 降低它的默认干扰
- 把它组织成更适合用户理解的 inspector

### 12.3 不要丢掉线程模型

Biomni 的任务模型和 BioClaw 的线程模型并不完全相同。改版时应该：

- 保留 thread 作为底层对象
- 在界面上把它“翻译”为任务卡
- 第一版不虚构“待办 / 进行中 / 已完成”这类后端尚未提供的 thread 状态

### 12.4 先做布局，再做细节

最先要解决的是：

- 信息架构
- 栏位主次
- 视觉焦点

而不是先纠结圆角、阴影和按钮颜色。

## 13. 验收标准

如果改版成功，应该满足下面这些判断标准：

### 视觉上

- 用户第一眼知道“当前主要内容在中间”
- 左右两栏都在辅助中心，而不是抢视觉中心
- 页面更像工作台，不像聊天 demo

### 使用上

- 用户能更快找到当前任务、当前结果和当前步骤
- 上传文件、查看结果、继续提问这三件事在同一工作流里完成
- PDF / 图片 / markdown 结果更容易被查看与复用

### 技术上

- 线程、消息、trace、upload 这些现有能力继续可用
- 不依赖后端重写
- 可分阶段上线

---

## 14. 最终建议

我建议本次改版不要理解成“把 BioClaw 的 CSS 调得像 Biomni”，而应理解成：

**把 BioClaw 本地网页从一个多面板聊天工具，升级成一个以任务成果为中心的分析工作台。**

真正应该学习 Biomni 的，是这三件事：

- 主舞台优先
- 左右分工明确
- 项目、步骤、结果围绕当前任务组织

如果按这个方向实施，BioClaw 不需要完全复制 Biomni，也能在保持自己能力的同时，达到非常接近的专业工作台体验。
